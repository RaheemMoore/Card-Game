# Archetype Weapon and Companion Reference

**Status:** Draft reference for approval  
**Project:** Card Game  
**Scope:** Canonical weapon families for all 11 archetypes, plus approved companion terminology  
**Source priority:** Latest project snapshot and current design direction

---

## 1. Purpose

This document defines the fantasy weapon families that may be associated with each archetype. It also prevents unrelated companion systems from being collapsed into a generic “summon” or “servant” category.

These entries are **weapon concepts and generation categories**, not finalized combat mechanics. A weapon appearing here does not automatically establish its stats, abilities, rarity, acquisition method, or animation requirements.

---

## 2. Canonical Companion Terminology

| Archetype | Companion Category | Relationship |
|---|---|---|
| Barbarian | None by default | Allies are clan members, peers, or followers—not equipment |
| Monk | None by default | Students, mentors, and temple allies remain people |
| Beastmaster | **Bonded Beasts** | Cooperative partnership built on trust |
| Druid | None by default | Temporary natural manifestations may be abilities, not permanent companions |
| Necromancer | **Undead Servants** | Raised, bound, inherited, contracted, or voluntarily retained dead |
| Vampire | **Blood Servants** | Sworn, enthralled, transformed, constructed, or politically subordinate beings |
| Lycanthrope | None by default | Pack members are peers, kin, or allies |
| Mech Pilot | **Controlled Units** | Machines directly piloted, remotely commanded, or tactically controlled |
| Android | **Autonomous Units** | Robots, drones, sub-AIs, and automated systems capable of independent action |
| Seraph | None by default | Celestial beings are peers, authorities, or allies |
| Human | None by default | Followers and teammates remain people |

### Critical distinctions

- **Necromancer:** The dead answer the character’s call.
- **Vampire:** Beings serve a bloodline, court, pact, or dominating will.
- **Beastmaster:** The animal fights alongside the character as a partner.
- **Mech Pilot:** The character actively controls the machine.
- **Android:** The machine or AI can operate independently.
- **Lycanthrope:** Pack members are never classified as servants.

# 3. Archetype Weapon Libraries

## Barbarian

**Core fantasy:** Carry a living legacy through hardship.

Barbarian weapons should feel inherited, repaired, culturally specific, and connected to survival, duty, or community. Oversized brutality may appear, but it should not replace the archetype’s identity.

| Weapon | Type | Fantasy role |
|---|---|---|
| Ancestor Blade | Longsword or greatsword | A repaired heirloom carrying the history of previous bearers |
| Oath Axe | One- or two-handed axe | Engraved with promises, clan debts, victories, or names |
| Hearth Hammer | Warhammer | A smithing or building tool transformed into a guardian’s weapon |
| Guardian Spear | Spear | Protects settlements, travelers, borders, and sacred ground |
| Legacy Shield | Shield | Layered with family marks, repairs, and memorial inscriptions |
| War Pick | Piercing weapon | A practical answer to armor, stone, and hostile terrain |
| Hunter’s Cleaver | Heavy blade | Combines survival utility with battlefield force |
| Clan Bow | Bow | A regional hunting weapon shaped by cultural craftsmanship |
| Stone Maul | Maul | Symbolizes endurance, place, and responsibility |
| Oath-Knot Flail | Flail | Chains and cords represent communal obligations |

**Recommended foundation set:** Ancestor Blade, Oath Axe, Guardian Spear, Hearth Hammer, Legacy Shield.

# 4. System Rules

## 4.1 Weapon classification

Each final weapon entry should eventually support these fields:

- Archetype
- Weapon family
- Weapon name
- Grip or equipment slot
- Damage or utility role
- Range
- Visual silhouette
- Material language
- Rank evolution
- Compatible abilities
- Prompt keywords
- Negative prompt rules
- Animation requirements
- Companion interaction, where applicable

These fields are recommendations for a future structured schema. They are not yet approved implementation requirements.

## 4.2 Rank evolution

The existing rank progression is:

1. **Foundation**
2. **Forged**
3. **Ascended**

Weapons should preserve their recognizable base identity across all three ranks.

- **Foundation:** Functional, grounded, minimally ornamented.
- **Forged:** Improved materials, clearer archetype identity, added construction detail.
- **Ascended:** Mythic expression, controlled supernatural effects, premium silhouette.

Rank evolution should enhance the weapon rather than replace it with an unrelated object.

## 4.3 Companion ownership

Companions must not be treated as cosmetic weapon attachments when they possess meaningful identity or agency.

- Bonded Beasts should have names, behavior, and partnership cues.
- Undead and Blood Servants may range from disposable units to recurring characters.
- Controlled Units remain mechanically dependent on a pilot command system.
- Autonomous Units may make independent decisions and should visually communicate intelligence or programmed purpose.

## 4.4 Visual generation safeguards

Avoid:

- Giving every archetype the same sword with a different glow color
- Turning every weapon into oversized glowing equipment
- Using servants for archetypes that do not support servant mechanics
- Calling Beastmaster animals “pets,” “minions,” or “servants”
- Calling Lycanthrope pack members servants
- Treating Mech Pilot drones and Android autonomous systems as identical
- Replacing Human identity with generic medieval equipment only
- Making Seraph weapons automatically benevolent or morally flawless
- Making every Necromancer servant skeletal
- Making every Vampire servant a mindless thrall

For the first implementation pass, create only the five foundation weapon families listed for each archetype. This produces:

- 11 archetypes
- 5 foundation weapon families per archetype
- **55 initial weapon families**

- 5 Bonded Beast types
- 8 Necromancer Undead Servant types
- 8 Vampire Blood Servant types

This is broad enough to test variety without immediately creating an unmanageable asset and prompt library.
